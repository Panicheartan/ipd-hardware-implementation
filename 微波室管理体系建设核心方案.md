# 微波室管理体系建设核心方案

## 基于"职责分工+工作流程+绩效考核"三要素

**版本**：V1.0  
**核心原则**：先明确"谁负责"，再明确"怎么配合"，最后明确"如何评价"  
**实施周期**：90天（第1-3个月）

---

## 第一章 核心逻辑

### 1.1 管理体系三要素

```
┌─────────────────────────────────────┐
│         第一要素：职责分工           │
│         （Who - 谁负责什么）         │
│                                     │
│    回答：每个岗位的职责是什么？      │
│          与其他岗位的协作关系？      │
└───────────────┬─────────────────────┘
                ↓
┌─────────────────────────────────────┐
│         第二要素：工作流程           │
│         （How - 工作怎么流转）       │
│                                     │
│    回答：工作从哪里开始？到哪里结束？│
│          中间经过哪些环节？谁审批？  │
└───────────────┬─────────────────────┘
                ↓
┌─────────────────────────────────────┐
│         第三要素：绩效考核           │
│         （Result - 如何评价结果）    │
│                                     │
│    回答：做到什么程度算好？算不好？  │
│          做好了怎么奖？做不好怎么罚？│
└─────────────────────────────────────┘
```

### 1.2 为什么这三要素是关键基础？

| 要素 | 如果没有会怎样？ | 建设目标 |
|:-----|:----------------|:---------|
| **职责分工** | 推诿扯皮，出问题找不到责任人 | 人人清楚"我是谁、我该做什么" |
| **工作流程** | 协作混乱，工作卡壳，效率低下 | 工作顺畅流转，减少内耗 |
| **绩效考核** | 干多干少一个样，优秀员工流失 | 奖优罚劣，激发团队动力 |

### 1.3 三要素之间的关系

**逻辑顺序**：职责分工 → 工作流程 → 绩效考核

1. **先定职责**：明确每个岗位"该做什么"
2. **再定流程**：明确工作"怎么配合完成"
3. **最后定考核**：明确"做到什么程度"如何评价

**相互支撑**：
- 职责清晰了，流程才能设计
- 流程顺畅了，考核才有依据
- 考核到位了，职责和流程才能落地执行

## 第二章 第一要素：职责分工

### 2.1 微波室组织架构

```
微波室（25人 + 主任/副主任）
├── 产品管理组（8人）
│   ├── 组长（1人）- 组内统筹、项目管理
│   ├── 研发管理（3人）- 需求分析、项目进度、客户对接
│   └── 可靠性管理（4人）- 可靠性/EMC/元器件/环境评估
│
├── 产品开发组（4人）
│   ├── 组长（1人）- 组内统筹、架构把关
│   └── 产品工程师（3人）- 探测①/探测②/通信产品开发
│
└── 技术开发组（11人）
    ├── 组长（1人）- 组内统筹、技术规划、对外对接
│   ├── 天线专业（3人）- 通信类/探测无源/探测有源天线
│   ├── 射频专业（3人）- 接收机/发射机/频率源
│   ├── 数字专业（2人）- FPGA/RFSOC
│   └── 电源专业（2人）- 二次电源（为各专业供电）
```

### 2.3 组间协作职责边界

#### 产品开发组 ↔ 技术开发组

| 协作内容 | 产品开发组职责 | 技术开发组职责 |

| **进度协调** | 跟踪进度，反馈变更 | 按计划开发，及时反馈风险 |
| **验收确认** | 验收模块，确认满足需求 | 交付模块，提供技术支持 |
| **问题处理** | 反馈使用问题 | 解决问题，优化模块 |

**协作接口**：产品开发组 → 技术开发组组长 → 各专业工程师

| 协作内容 | 产品管理组职责 | 产品开发组职责 |
|:---------|:--------------|:--------------|
| **需求输入** | 转化客户需求为标准化规格 | 接收规格，提出疑问 |
| **进度管理** | 跟踪项目进度，协调资源 | 按计划执行，反馈偏差 |
| **DFX评估** | 组织可靠性/EMC/环境评估 | 配合评估，整改问题 |
| **交付验收** | 组织交付评审，移交生产 | 交付设计资料，支持批产 |

## 第三章 第二要素：工作流程

### 3.1 核心工作流程清单

| 流程名称 | 目的 | 主要环节 | 参与方 |
|:---------|:-----|:---------|:-------|
| **组间协作流程** | 产品开发组与技术开发组高效协作 | 需求提出→评估反馈→方案制定→开发交付→验收确认 | 产品开发组、技术开发组 |
| **需求管理流程** | 需求清晰可追溯，减少返工 | 需求收集→分析确认→变更控制→实现验收 | 产品管理组、产品开发组 |
| **IPD阶段评审流程** | 确保质量，避免后期返工 | CDCP→PDCP→开发评审→TR评审→ADCP | 全组参与 |
| **周协调会机制** | 定期对齐，及时解决问题 | 需求更新→进度汇报→问题协调→计划确认 | 产品开发组长、技术开发组长 |
| **技术分享机制** | 知识传承，团队能力提升 | 分享准备→现场分享→讨论交流→文档入库 | 全员 |

### 3.2 组间协作流程（产品开发组 ↔ 技术开发组）

```
产品开发组                        技术开发组
    │                                  │
    │ ① 提出模块需求                   │
    │ （性能指标、交付时间）            │
    │─────────────────────────────────→│
    │                                  │
    │                                  │ ② 需求评估
    │                                  │ （可行性、工作量、风险）
    │←─────────────────────────────────│
    │ ③ 评估反馈                       │
    │ （是否可行、预计交付时间）        │
    │                                  │
    │ ④ 参与方案评审                   │
    │─────────────────────────────────→│
    │                                  │ ⑤ 制定技术方案
    │←─────────────────────────────────│
    │ ⑥ 方案确认                       │
    │                                  │
    │ ⑦ 跟踪进度、反馈变更              │←→│ ⑧ 按计划开发、及时反馈风险
    │                                  │
    │ ⑨ 验收模块                       │
    │─────────────────────────────────→│
    │←─────────────────────────────────│
    │ ⑩ 交付模块、提供技术支持          │
```

**协作接口人**：
- 产品开发组 → 产品开发组组长 → 具体产品工程师
- 技术开发组 → 技术开发组组长 → 各专业工程师

**时效要求**：
- 需求评估反馈：2个工作日内
- 技术方案制定：5个工作日内
- 进度风险反馈：发现后立即反馈

### 3.3 周协调会机制

| 要素 | 内容 |
|:-----|:-----|
| **时间** | 每周三下午 14:00-15:00 |
| **参与人** | 产品开发组长、技术开发组长 |
| **议程** | 需求更新（5分钟）→ 进度汇报（10分钟）→ 问题协调（10分钟）→ 计划确认（5分钟） |
| **产出** | 会议纪要 + 行动项（责任人+完成时间） |
| **主任参与** | 每月参加1次 |

### 3.4 需求管理流程

| 阶段 | 动作 | 责任人 | 交付物 |
|:-----|:-----|:-------|:-------|
| **需求收集** | 收集客户需求、内部需求 | 产品管理组 | 需求清单 |
| **需求分析** | 分析可行性、工作量、风险 | 产品管理组+产品开发组 | 需求分析报告 |
| **需求确认** | 与客户确认需求规格 | 产品管理组 | 需求规格说明书（SRS） |
| **需求变更** | 评估变更影响，审批变更 | 变更提出人+项目经理 | 变更请求表（CR） |
| **需求实现** | 按计划开发实现 | 产品开发组+技术开发组 | 设计文档、代码 |
| **需求验收** | 验证满足需求 | 产品管理组 | 验收报告 |

**需求台账管理**：每个需求必须录入台账，跟踪状态

### 3.5 技术分享机制

| 要素 | 内容 |
|:-----|:-----|
| **时间** | 每周五下午 16:00-17:00 |
| **参与人** | 全员（25人） |
| **形式** | 轮流分享，30分钟分享 + 30分钟讨论 |
| **排班** | 各组轮流，每月排班表 |
| **要求** | 分享前3天提交PPT审核，分享后1天文档入库 |
| **激励** | 分享积分纳入绩效考核，季度评选最佳分享奖 |

### 3.6 IPD阶段评审流程

| 评审点 | 目的 | 参与人 | 交付物 |
|:-------|:-----|:-------|:-------|
| **CDCP**（概念决策） | 确认需求和技术可行性 | 主任主持，全员参与 | 评审报告、决策结论 |
| **PDCP**（计划决策） | 确认方案和资源配置 | 主任主持，全员参与 | 评审报告、项目计划 |
| **TR**（技术评审） | 开发过程中技术把关 | 组长主持，相关专家 | 评审报告、问题清单 |
| **ADCP**（发布决策） | 确认质量和交付readiness | 主任主持，全员参与 | 评审报告、交付物 |

## 第四章 第三要素：绩效考核

### 4.1 考核原则

| 原则 | 说明 |
|:-----|:-----|
| **差异化** | 三组（技术开发/产品开发/产品管理）考核维度不同 |
| **量化为主** | 尽量用数据说话，减少主观评价 |
| **及时反馈** | 月度跟踪，季度面谈，及时发现问题 |
| **激励导向** | 奖励优秀，帮助改进，不是惩罚工具 |

### 4.2 三组差异化考核

#### 技术开发组（11人）

| 考核维度 | 权重 | 关键指标 | 目标值 |
|:---------|:----:|:---------|:------:|
| **技术成果** | 40% | 模块按时交付率 | ≥95% |
| | | 模块性能达标率 | ≥95% |
| **项目支持** | 30% | 需求响应及时率 | ≥95% |
| | | 技术支持满意度 | ≥4.0/5.0 |
| **技术积累** | 20% | 知识文档贡献 | ≥2篇/季度/人 |
| | | 技术分享次数 | ≥1次/季度/人 |
| **协作配合** | 10% | 协作满意度 | ≥4.0/5.0 |

#### 产品开发组（4人）

| 考核维度 | 权重 | 关键指标 | 目标值 |
|:---------|:----:|:---------|:------:|
| **产品成果** | 50% | 产品按时交付率 | ≥90% |
| | | 产品一次通过率 | ≥85% |
| **架构设计** | 25% | 架构评审通过率 | ≥90% |
| | | 接口定义准确率 | ≥95% |
| **跨组协作** | 25% | 协作满意度 | ≥4.0/5.0 |

#### 产品管理组（8人）

| 考核维度 | 权重 | 关键指标 | 目标值 |
|:---------|:----:|:---------|:------:|
| **项目成功** | 40% | 项目按时交付率 | ≥90% |
| | | 客户满意度 | ≥4.2/5.0 |
| **DFX质量** | 35% | 评估准确率 | ≥90% |
| | | 问题发现率 | ≥90% |
| **协调能力** | 25% | 问题响应时间 | ≤4小时 |

### 4.3 绩效管理周期

| 周期 | 动作 | 参与人 |
|:-----|:-----|:-------|
| **季度初** | 签订绩效目标 | 上级+员工 |
| **每月** | 绩效跟踪反馈 | 上级+员工 |
| **季度末** | 绩效考核评估 | 上级+员工 |
| **季度末** | 绩效面谈 | 上级+员工 |

### 4.4 绩效结果应用

| 等级 | 比例 | 结果应用 |
|:-----|:----:|:---------|
| **S**（卓越） | ≤10% | 晋升优先、大幅调薪、表彰 |
| **A**（优秀） | ≤20% | 优先调薪、发展机会 |
| **B+**（良好） | ~35% | 正常调薪 |
| **B**（合格） | ~25% | 维持现状，帮助改进 |
| **C**（待改进） | ~10% | 绩效改进计划（PIP） |
